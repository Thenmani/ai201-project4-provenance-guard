# Provenance Guard — Planning

A pluggable backend that lets any creative-sharing platform classify whether
submitted text was written by a human or generated by AI, score confidence in
that classification, surface a plain-language transparency label, and let
creators appeal a verdict they believe is wrong.

Scope of this document: All the required features along with a stretch feature, Ensemble detection (voting)

---

## Overview

The hard problem here is **not** raw accuracy — perfect AI detection is unsolved.
The engineering challenge is communicating *calibrated uncertainty* honestly and
giving misclassified creators a real path to contest a verdict.

One asymmetry drives the whole design:

> **A false positive — labeling a human's work as AI-generated — is worse than a
> false negative.** On a writing platform, a false accusation damages a real
> creator's reputation. The system is therefore deliberately biased to require
> *more* evidence before asserting "AI" than before asserting "human," and to
> route borderline cases into an honest "uncertain" verdict rather than a
> confident accusation.

This single principle shapes the confidence→label mapping, threshold placement,
label wording, and the appeals design throughout.

## Architecture

### Components

| Component | Responsibility |
|---|---|
| **API layer (Flask)** | Receives requests, attaches rate limiting, routes to handlers. |
| **Input validation / preprocessing** | Confirms text is present and within length bounds, assigns a `content_id`, normalizes whitespace. |
| **Signal 1 — LLM detector (Groq)** | Sends text to `llama-3.3-70b-versatile`, parses back an AI-likelihood score plus a one-line rationale. *(semantic / holistic)* |
| **Signal 2 — Stylometric analyzer (pure Python)** | Computes structural features (sentence-length variance, type-token ratio, punctuation density) and maps them to a sub-score. *(structural / statistical)* |
| **Confidence fusion** | Combines the two signals into one calibrated `ai_likelihood` and a verdict, applying the false-positive asymmetry. |
| **Label generator** | Maps the `ai_likelihood` band to one of three transparency label texts. |
| **Audit logger** | Writes the full structured decision record (signals, scores, label, status, appeals). |
| **Persistence store (SQLite or JSON)** | Holds the content record and its status so appeals can find it later. |
| **Appeals handler** | Looks up the original decision, appends creator reasoning, flips status to `under_review`. |
| **Response builder** | Assembles the JSON returned to the platform. |

### Narrative — submission and appeal flows

On **submission**, a creator's text hits `POST /submit`, passes the rate limiter
and validation, then fans out to the LLM detector and the stylometric analyzer;
their two scores are fused into one calibrated `ai_likelihood`, mapped to a
verdict and a transparency label, written to the audit log, and returned to the
caller. On **appeal**, a creator calls `POST /appeal` with the `content_id` and
their reasoning; the appeals handler looks up the original decision, records the
reasoning, flips the content's status to `under_review`, logs the appeal beside
the original decision, and confirms the new status — no automated
re-classification is performed.

## Detection Signals

The pipeline uses **two genuinely distinct signals**: one semantic, one
structural. They fail in *different* ways, which is exactly why combining them is
more informative than either alone. Every signal has blind spots — naming them is
part of understanding the signal.

### Signal 1 — LLM classification (Groq) — *semantic / holistic*

- **Measures:** whether the text reads as human- or AI-authored as a whole —
  voice, idiosyncrasy, and the over-smooth transitions and generic, hedged,
  evenly-balanced phrasing that machine text tends toward.
- **Why it differs between human and AI:** AI prose carries distributional
  fingerprints — low risk-taking, a consistent register, and an absence of the
  weird, specific choices humans make.
- **Output shape:** the model is prompted to return an estimated probability
  (0–100) that the text is AI-generated, plus a one-line rationale. This is
  normalized to **`llm_score ∈ [0,1]`**.
- **Blind spot:** it is a model judging models. It can be fooled by lightly
  human-edited AI text, and it can wrongly flag *polished or formal human
  writing* — academic, corporate, or ESL writers — as AI. It has no ground truth
  (it is a vibe with a number attached) and can drift run-to-run.
  **This is the primary false-positive source.**

### Signal 2 — Stylometric heuristics (pure Python) — *structural / statistical*

- **Measures:** quantifiable variance in the text:
  - **sentence-length variance** (burstiness),
  - **type-token ratio** (vocabulary diversity),
  - **punctuation density / variety**.
- **Why it differs between human and AI:** human writing is bursty and uneven
  (short sentence, long sentence, fragment); AI text clusters around uniform
  mid-length sentences with smoother distributions.
- **Output shape:** each feature is mapped to a 0–1 "AI-likeness" sub-value
  (more uniform / less diverse = more AI-like), and the sub-values are combined
  with a **burstiness-dominant weighting (0.70 / 0.15 / 0.15)** into
  **`stylo_score ∈ [0,1]`**. (See the Calibration note below for why weighting,
  not a flat average.)
- **Blind spot:** it is blind to meaning and easily gamed — an adversary can
  inject variance on purpose. It needs length to be reliable, so it is
  near-useless on a four-line poem or a short excerpt, and it penalizes humans
  who legitimately write uniformly (technical writers, strict poetic forms, ESL).

### Independence claim

One signal is **semantic**, the other is **structural**. They do not share a
failure mode, so when they *disagree*, that disagreement is itself information
(it pushes toward "uncertain"); when they agree, confidence is better founded
than either signal could justify alone.

### Fusion — combining the two signals

Both signals output the same axis (`0` = human-like, `1` = AI-like), so they
combine on a common scale. The fused score is a weighted average, **LLM-weighted
because stylometry is the noisier signal**:

```
ai_likelihood = 0.65 × llm_score + 0.35 × stylo_score
```

Two asymmetry-protecting rules override the raw average:

1. **Disagreement → uncertain.** If `|llm_score − stylo_score| > 0.40`, the
   signals contradict each other; rather than trust either, clamp
   `ai_likelihood` into the Uncertain band (see below). Disagreement is a reason
   to *withhold* a confident verdict.
2. **Short-text guard.** Below ~40 words, stylometry is unreliable. Drop its
   weight to near-zero (rely on the LLM) and widen the Uncertain band so the
   system is slower to make any confident claim on short text.

## Uncertainty Representation

### The single internal axis

The system reasons on one number: **`ai_likelihood ∈ [0,1]`** — its estimated
probability that the text is AI-generated.

- `0.0` → certainly human
- `1.0` → certainly AI
- `0.5` → a genuine coin flip; the system has no idea

So **a score of 0.6 means "leaning AI, but not confidently"** — it falls inside
the Uncertain band and produces *no* accusation. This is the heart of the design:
0.6 is not "AI," it is "we won't claim."

### Asymmetric bands (where the false-positive principle lives)

The verdict is derived from `ai_likelihood` using **deliberately asymmetric**
cutoffs. The bar to accuse ("Likely AI") is set high and narrow; the
"Uncertain" band is wide and skewed toward the AI side, so borderline-AI text is
caught as *uncertain* rather than falsely accused.

| `ai_likelihood` | Verdict | Notes |
|---|---|---|
| `≥ 0.75` | **Likely AI** | Narrow band — hard to reach by design; requires both signals to agree. |
| `0.40 – 0.75` | **Uncertain** | Wide band, skewed toward the AI side. |
| `≤ 0.40` | **Likely Human** | |

(Short-text submissions, < 40 words, use a wider uncertain band: `≥ 0.82`
Likely AI, `≤ 0.35` Likely Human. See Calibration note and the short-text guard.)

### Confidence reported to the user

The single axis is translated into a human-facing confidence depending on the
verdict:

- **Likely AI:** `confidence = ai_likelihood` (e.g. 0.92 → "92% confidence").
- **Likely Human:** `confidence = 1 − ai_likelihood` (e.g. 0.10 → 90% confidence).
- **Uncertain:** no numeric confidence is asserted; the label communicates that
  no claim is being made.

This is why **0.51 and 0.95 produce meaningfully different labels** — they fall
in different bands (Uncertain vs. Likely AI) and carry different confidence text,
rather than flipping at a single 0.50 boundary.

### Calibration — how I test that scores are *meaningful*

A score is only meaningful if it *separates* known cases. Test procedure
(executed in Milestone 4):

1. Assemble ~5 known-human passages (varied: casual, formal, ESL-style, a poem)
   and ~5 known-AI passages (varied prompts/models).
2. Run all through the pipeline and record `ai_likelihood`.
3. **Success criterion:** human samples land `≤ 0.40` (Likely Human) and AI
   samples land `≥ 0.75` (Likely AI), with genuinely ambiguous samples landing in
   the Uncertain band rather than being forced to a pole.
4. If human samples creep above 0.40 (false positives), raise the AI cutoff
   and/or lower the LLM weight — the asymmetry must hold.

### Calibration findings (Milestone 4)

Running the pipeline on the test corpus produced two deliberate adjustments to the
original spec. Both are recorded here so the code and this document stay in sync.

1. **Stylometry: burstiness-dominant weighting, not a flat average.** Exploration
   showed that on short samples, type-token ratio clustered tightly (≈0.83–0.90 for
   every sample, human or AI) and punctuation variety was noisy — neither separated
   the cases. Sentence-length burstiness (coefficient of variation) was the only
   sub-metric that cleanly separated human / AI / formal text. A flat average would
   have let the two dead metrics wash out the one that works, so the sub-scores are
   combined **0.70 burstiness / 0.15 diversity / 0.15 punctuation**. This is also a
   concrete instance of the stylometry blind spot ("needs length to be reliable").

2. **AI cutoff lowered 0.80 → 0.75 (short-text 0.85 → 0.82).** With the LLM
   realistically topping out around 0.80–0.90 and stylometry around 0.70, the
   maximum fused score for genuine AI text sat near 0.80 — so a 0.80 cutoff made the
   *Likely AI* band practically unreachable even when both signals agreed. 0.75 makes
   the band reachable when (and only when) both independent signals agree, while
   still sitting well above the 0.62 that the formal-human false-positive trap
   produced — so the asymmetry and the human protection both hold. Verified:
   the productivity-cliché sample (LLM 0.80, stylo 0.70 → fused 0.765) reaches
   *Likely AI*; the formal-human academic sample (fused 0.62) stays *Uncertain*.

## Transparency Label Design

Three variants, written verbatim (these are also reproduced in the README, where
they are graded). Note the asymmetry in *tone*: the AI label is hedged,
non-accusatory, and explicitly names the appeal path; the Uncertain label refuses
to guess.

**High-confidence AI**
> ⚠️ **Likely AI-Generated.** Our automated analysis suggests this text was
> probably produced with AI (estimated confidence: 92%). This is a statistical
> signal, not a certainty, and not an accusation. If you wrote this yourself, you
> can appeal this assessment.

**High-confidence human**
> ✓ **Likely Human-Written.** Our automated analysis found no strong signs of AI
> generation (estimated confidence: 90%). This is an estimate, not a guarantee of
> authorship.

**Uncertain**
> ❓ **Inconclusive.** We couldn't determine with confidence whether this was
> written by a person or generated by AI. Rather than guess, we're telling you
> plainly: no attribution claim is being made here.

> The confidence percentages shown above are illustrative; at runtime each label
> interpolates the actual reported confidence for that submission.

project and confirm they correctly understand what each one is claiming — before

## Appeals Workflow

- **Who can appeal:** the content owner. In production this is an authenticated
  creator; for this project, anyone holding the `content_id` returned at
  submission.
- **What they provide:** `content_id` and a free-text `reasoning` field stating
  their case for why the classification is wrong.
- **What the system does on receipt:**
  1. Validate that the `content_id` exists (`404` if unknown).
  2. Append an appeal record (reasoning + timestamp) to that content's entry.
  3. Update status `classified → under_review`.
  4. Write an `appeal` event to the audit log, referencing the original decision
     (verdict, confidence, signals).
  - **No automated re-classification** — a human reviewer owns the next step.
- **What a human reviewer sees** (the appeal queue, e.g. `GET /appeals` or the
  `under_review` items in `/log`): for each item — `content_id`, original verdict
  and confidence, the per-signal breakdown (`llm_score`, `stylo_score`), a text
  excerpt, the creator's reasoning, and submission/appeal timestamps.

## Edge Cases

Specific scenarios this system is expected to handle poorly, with mitigations:

1. **Repetitive, simple-vocabulary poetry** (a villanelle, a nursery-style
   rhyme). Deliberate repetition drives the type-token ratio down and sentence
   structure is uniform, so **stylometry scores it as strongly AI-like** even
   though it is a legitimate human form.
   *Mitigation:* the short-text guard reduces stylometry's weight on short
   pieces, the LLM signal carries more weight, and the disagreement→uncertain
   rule catches cases where the LLM disagrees with stylometry.

2. **ESL or highly formal / templated human writing** (cover letters, academic
   abstracts, corporate copy). Smooth, even, hedged prose that *both* signals can
   misread as AI — the core false-positive trap, where the two signals agree and
   are both wrong.
   *Mitigation:* the asymmetric thresholds keep borderline cases in Uncertain
   rather than Likely AI; the non-accusatory label wording; and a frictionless
   appeal path.

3. *(adversarial)* **Lightly human-edited AI text** — burstiness and odd word
   choices injected on purpose to defeat stylometry, with enough editing to
   soften the LLM signal. Honest stance: detection is not perfect, which is
   precisely why the appeals path and the "Uncertain" verdict exist.

# Stretch feature — Ensemble detection (voting)

## What I'm adding and why

Right now the system uses two signals (the Groq LLM and the stylometry heuristics)
and blends their scores with a weighted average. For the ensemble stretch feature
I'm adding a third signal and changing how the signals are combined: instead of
averaging numbers, each signal will cast a vote, and the final verdict comes from
counting the votes.

I picked voting because it's the clearest way to show "ensemble" thinking — three
independent opinions that have to agree before the system commits to a confident
answer. It also fits the thing I care about most in this project: I'd rather the
system say "we're not sure" than wrongly accuse a real writer. With voting, if the
three signals don't agree, that disagreement naturally becomes an "uncertain"
result instead of a confident guess.

## The third signal — repetition / redundancy (pure Python)

The two signals I already have look at different things: the LLM reads the meaning
and overall feel of the text, and the stylometry looks at structure like sentence
length variation. For the third signal I want something genuinely different from
both, otherwise it's not really a third opinion.

So the third signal measures **repetition**: how much the text repeats itself.
AI writing often leans on the same phrases and the same transition words over and
over ("Furthermore", "It is important to note", "Additionally"), and reuses the
same sentence openings. Human writing tends to repeat less mechanically. I'll
measure this in plain Python by looking at things like repeated word pairs and
triples, and how varied the sentence beginnings are. A more repetitive text gets a
higher AI-likeness score.

I chose a pure-Python heuristic instead of a second LLM call on purpose. A second
LLM call would cost another API request every time and, more importantly, it would
probably agree with the first LLM most of the time (same model, same blind spots) —
so it wouldn't really be an independent third vote. The repetition heuristic is
free to run and looks at something neither of the other two signals checks directly.

**What it measures:** how much the text repeats words, phrases, and sentence
openings.

**Why it differs between human and AI:** AI text tends to reuse phrasing and
transitions more mechanically; human writing varies more.

**Its blind spot:** some legitimate human writing is deliberately repetitive —
poetry with refrains, song-style lyrics, rhetorical repetition for effect. Those
could look "AI-like" to this signal. That's exactly why it's only one vote out of
three and can be outvoted by the other two.

## How the voting works

Each of the three signals turns its score into one of three votes:

- **AI** — this signal thinks the text is AI-generated
- **human** — this signal thinks the text is human-written
- **abstain** — this signal doesn't trust its own answer here, so it sits this one out

The "abstain" option is important and is where my existing safety rules now live
naturally. For example, the stylometry signal isn't reliable on very short text, so
on short submissions it abstains instead of voting — that's the old "short-text
guard" built right into the voting model. A signal can also abstain when its score
is right in the middle (it genuinely can't tell).

Then I count the votes, and the rules are **deliberately asymmetric** because a
false accusation is worse than a missed one:

- To return **likely_ai**, I want real agreement — at least two signals voting AI
  and *no* signal voting human. One lonely AI vote is not enough to accuse someone.
- To return **likely_human**, the bar is lower — a simple majority of human votes
  is enough, because clearing someone is the safer direction to be wrong in.
- Anything else — a split, lots of abstentions, or AI votes that are contradicted
  by a human vote — comes back as **uncertain**.

This means my old "if the signals disagree, say uncertain" rule isn't a special
case bolted on anymore; it's just what happens when the votes are split.

## Keeping the confidence number meaningful

I don't want to lose the calibrated confidence I worked hard on in Milestone 4
(where 0.62 means "uncertain" and 0.77 means "confidently AI"). Pure vote-counting
would make confidence blunt — basically just "3 out of 3" or "2 out of 3."

So I'm keeping a confidence number alongside the votes, and I'll base it on **how
decisive the vote was**:

- All three signals agree → high confidence
- Two agree, one abstains → medium-high confidence
- A close or split vote → low confidence (which is what "uncertain" should feel like)

That way the votes decide the verdict, but the confidence still reflects how strong
the agreement was — so the three transparency labels keep showing a meaningful
percentage, not just a tally.

## Showing the votes to the user

To make the ensemble visible (and honest), I'll include each signal's individual
vote in the API response and in the audit log, something like:

```
votes: { llm: "ai", stylometry: "human", repetition: "ai" }  ->  verdict: uncertain
```

This shows the "three opinions deliberating" rather than hiding it behind one
number, and it makes the audit log more useful for a human reviewer reading an appeal.

## What changes and what doesn't

**Changes:** I add one new signal file for the repetition heuristic, and I rewrite
the scoring logic so it votes and tallies instead of averaging. The scoring rewrite
is the bulk of the work.

**Stays the same:** the `/submit`, `/appeal`, and `/log` endpoints, the three
transparency labels, the audit log format (just gains the votes), rate limiting,
and the existing LLM and stylometry signals. I'm only touching the scoring core and
adding one signal.

## Re-calibration (the real work)

Adding a third voter changes every result, so the main job isn't writing the code —
it's re-testing. After it's built I'll re-run my calibration set (`test_pipeline.py`,
extended to print the votes) and check three things:

1. Clearly-human text still comes back **likely_human**.
2. Clearly-AI text where signals agree still reaches **likely_ai**.
3. The formal / academic human sample — my main false-positive worry — still lands
   in **uncertain** and is never confidently accused.

If the third signal pushes any human-written sample toward a confident AI verdict,
I'll tighten the voting rule (for example, require the AI vote to be unanimous) or
make the repetition signal abstain more readily. The asymmetry — protecting real
writers from false accusations — has to survive the change, or the feature isn't worth shipping.
